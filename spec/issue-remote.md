issue 管理完整方案

核心目标

通过 issue-manager skill 统一管理 GitHub issue 的生命周期，支持并行处理多个 issue。

---
两个组件

1. issue-manager skill

路径：skills/issue-manager/

印记文件：.claude/issue-manager/active/issue-{N}-{slug}.yaml

命令接口：

┌───────────────┬─────────────────────────────────────────────────────────────────────────────────────────────┐
│     命令      │                                            行为                                             │
├───────────────┼─────────────────────────────────────────────────────────────────────────────────────────────┤
│ pull <id>     │ fetch origin → 检查未提交改动 → git checkout -b issue-{N}-{slug} origin/main → 创建印记文件 │
├───────────────┼─────────────────────────────────────────────────────────────────────────────────────────────┤
│ list          │ 列出所有 in_progress 的 issues                                                              │
├───────────────┼─────────────────────────────────────────────────────────────────────────────────────────────┤
│ checkout <id> │ 切换到该 issue 的分支                                                                       │
├───────────────┼─────────────────────────────────────────────────────────────────────────────────────────────┤
│ status        │ 查看当前分支状态                                                                            │
├───────────────┼─────────────────────────────────────────────────────────────────────────────────────────────┤
│ publish <id>  │ 1. 身份校验2. 前置检查3. 测试关卡4. 用户确认5. 执行 push6. 创建 PR                          │
└───────────────┴─────────────────────────────────────────────────────────────────────────────────────────────┘

2. issue-reminder hook

路径：hooks/issue-reminder/

行为：会话启动时扫描印记目录，提醒正在处理的 issue 或提示开始新 issue

---
    git pre-command hooks

配置环境文件中的 PreCommand hooks，拦截：
- git fetch 前检查：不存在强制覆盖风险，可放行
- git checkout 前检查：提示或 block，避免误切换分支
- git pull 前检查：提示先 stash 或 commit

---
pull 命令前置检查（严格） 

- 检查当前工作区是否有未提交改动
- 检查当前分支有未合并的 commit
- 检测到上述情况 → abort，提示"存在未提交改动"

---
publish 命令完整流程

issue-manager publish <id>

步骤：

Step 1：身份校验
- 检查印记文件存在
- 检查当前分支匹配 issue-{N}-{slug} 模式

Step 2：前置检查
- 检查有相对 origin/main 的实质代码 commit

Step 3：测试关卡
- 运行相关测试
- 测试失败 → abort

Step 4：用户确认
- 提示即将 push 的分支、PR 标题、改动摘要
- 等待用户明确确认
- 用户拒绝 → abort

Step 5：执行 push
- git push origin issue-{N}-{slug}:issue-{N}-{slug}-pr

Step 6：创建 PR
- gh pr create --base issue-{N}-{slug}-pr

Step 7：收尾
- 标记印记文件或按用户偏好删除

关键设计点

│        点         │                      设计                       │
├───────────────────┼─────────────────────────────────────────────────┤
│ push 严格性       │ 4 道关卡（身份校验 → 前置检查 → 测试 → 确认）   │
├───────────────────┼─────────────────────────────────────────────────┤
│ pull 安全性       │ pull 前检查未提交改动，block 强制覆盖           │
├───────────────────┼─────────────────────────────────────────────────┤
│ 本地 issue 关联   │ 解耦，可选字段的 local_issue_file               │
├───────────────────┼─────────────────────────────────────────────────┤
│ 本地改动保护      │ git PreCommand hooks 提示或 block checkout/pull │
├───────────────────┼─────────────────────────────────────────────────┤
│ 远程 main 保护    │ PR 目标分支固定 issue-{N}-{slug}-pr             │
├───────────────────┼─────────────────────────────────────────────────┤
│ issue-solved hook │ 独立运行不受影响                                │
├───────────────────┼─────────────────────────────────────────────────┤
│ 并行支持          │ 多个 issue 同时在印记目录存在                   │
└───────────────────┴─────────────────────────────────────────────────┘
