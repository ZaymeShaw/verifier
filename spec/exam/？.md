从业务定义看必须实现扩展点

一、四个 agent 的角色定位 vs 必须实现扩展点

1. Live（被测系统具现化）→ build_request(case) -> Dict

Live 的角色是"靶子"——把外部输入转成业务系统真实接受的请求体。这个翻译动作是项目特有的：
- 每个项目的 API 形状不同（QA 是 question/contexts，client_search 是 user_text，marketing 是 query/turns）
- "case.input → normalized_request"的映射是 Live 的核心业务知识，不可能有通用默认值
- 如果默认返回 {}，Live 就无法投递任何请求——所以必须实现

2. Mock（扮演用户）→ scenarios() -> List[str]

Mock 的角色是"产用户侧输入"。场景列表是项目特有的业务语义：
- QA 的场景是 qa_gold_answer/qa_weak_quality（质量维度）
- client_search 的场景是按字段能力分的
- marketing 的场景是 intent/clarification/planning（阶段维度）
- 场景名直接驱动 mock_agent 生成不同用例，没有通用默认场景

3. Judge（业务评估）→ build_context(trace) -> Dict

Judge 的角色是"业务角度评估输出是否满足意图"。评估上下文是项目特有的判定逻辑：
- 每个项目的判定维度不同（QA 看 answer quality，client_search 看 conditions 字段匹配，marketing 看 path/cards）
- 上下文含 system_prompt_extras、user_prompt_extras、critical_dimensions 等项目特有字段
- 如果默认返回 {}，LLM 没有判定依据——所以必须实现

4. Attribute（代码归因）→ build_context(trace, judge_result) -> Dict

Attribute 的角色是"系统不及预期时从代码链路归因"。归因上下文是项目特有的业务链路：
- 每个项目的 business_chain 不同（QA 是 input_normalization→answer_generation，client_search 是 request_normalization→routing_pattern_match）
- 归因策略、tool 选择、证据契约都是项目特有的
- 如果默认返回 {}，LLM 无法定位根因——所以必须实现

二、角色定位决定必须实现扩展点的逻辑

四个角色的共同点：它们的核心业务知识无法通用化。

- Live 的"输入翻译"——每个项目 API 形状不同
- Mock 的"场景定义"——每个项目业务场景维度不同
- Judge 的"判定上下文"——每个项目评估维度不同
- Attribute 的"归因上下文"——每个项目业务链路不同

通用部分（流程骨架、LLM 调用机制、校验、fallback）已经在协议层模板方法里固化了。项目特有部分（业务知识）必须由项目实现。

这就是"必须实现"的判定原则：如果这个动作没有通用默认值，且缺失它会导致流程无法产出有效结果，它就必须是 abstractmethod。

三、如何定义类型确保可扩展性和上限

当前 4 个扩展点的返回类型都是 Dict[str, Any]，这有问题：


问题：
- Dict[str, Any] 太宽松，项目可以塞任何东西，但模板方法不知道有哪些字段可用
- 上限低——LLM 调用需要特定字段（如 system_prompt_extras、user_prompt_extras），但类型不约束
- 可扩展性看起来高（任意 Dict），实际上是"无约束"而非"高扩展"

改进方向：

1. Live.build_request 返回类型：应该返回 Dict[str, Any]（normalized_request），但约束"必须符合
live_schema.REQUEST_SCHEMA"。这个约束用类型表达不了，靠运行时 live_schema.check.request 校验。当前设计合理。
2. Mock.scenarios 返回类型：List[str] 足够，但可以约束"场景名必须在 live_schema.SCENARIO_ENUM 内"。同样靠运行时校验。
3. Judge/Attribute.build_context 返回类型：这是关键问题。当前 Dict[str, Any] 允许项目塞任何字段，但模板方法的 _run_llm_judge/_run_llm_attribute
需要特定字段（如 system_prompt_extras、user_prompt_extras、tool_call_limit）。

建议的改进：

为 Judge/Attribute 的 build_context 返回值定义一个结构化 dataclass（如 JudgeContext、AttributeContext），包含：
- 通用必填字段（如 system_prompt_extras: List[str]、user_prompt_extras: Dict）
- 项目扩展字段（extras: Dict[str, Any] 用于项目特有字段）

这样：
- 可扩展性：项目可以通过 extras 塞特有字段
- 上限高：通用字段有类型约束，模板方法能可靠使用
- 契约清晰：项目知道必须提供哪些字段

但这会增加项目实现的复杂度（要构造 dataclass 而非 Dict）。权衡：当前 Dict[str, Any] 灵活但松散；dataclass 严谨但重。考虑到 judge/attribute
上下文字段较多且项目差异大，Dict + 文档约定可能是合理的折中。

四、总结

必须实现的判定原则：核心业务知识无法通用化、缺失导致流程无效。

当前的 4 个选择是合理的：
- Live.build_request：输入翻译，项目特有
- Mock.scenarios：场景定义，项目特有
- Judge.build_context：判定上下文，项目特有
- Attribute.build_context：归因上下文，项目特有

类型定义的改进空间：
- Live/Mock 的返回类型约束靠 live_schema 运行时校验，合理
- Judge/Attribute 的 Dict[str, Any] 可考虑改为结构化 dataclass，提升契约清晰度和上限，但需权衡实现成本

上限提升的关键：不是把类型定得更严，而是把"通用字段"和"项目扩展字段"分清——通用字段有类型约束，项目字段用扩展机制（extras 或 mixin）。
