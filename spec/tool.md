节点逐个复现"，但拆解的对象应该是业务系统本身的 live 链路结构，trace 节点只是拆解后用来对照的样本。

修正后的逻辑应该是这样：

真正的拆解对象：业务系统的 live 链路

拆解不是"沿 trace 节点逐个跑"，而是：

1. 去业务系统里把 live 链路找出来——live
一次执行在业务系统内部经过了哪些阶段、每个阶段调了哪个原函数、上下游是怎么传参的。这是从业务系统源码/调用图里拆出来的，不是从 verifier 的 trace
里读出来的。
2. 把这条 live 链路拆成局部链路/函数——按业务系统内部的执行顺序，每个有确定性转换的环节就是一段局部链路。段之间有依赖（上一段的输出是下一段的输入
），但每段可独立重放（只要给输入就能跑）。
3. 每段局部链路对应一个"输入 schema → 业务系统函数 → 输出 schema"的闭环——这个闭环是业务系统自己定义的，verifier 只是把它搬过来作为模拟单元。
4. trace 节点是验证样本，不是拆解依据——verifier 的 trace.execution_trace 里记录的 live 节点，是这条业务系统 live
链路在某次具体执行时的快照。模拟时用 trace 节点的 evidence 当输入、用 trace 的 actual 当对照，但拆解的段划分来自业务系统链路本身，不来自 trace
节点列表。

拆解步骤的修正

之前的流程："读 trace 节点 → 对每个节点找业务函数复现"——这是本末倒置。

正确流程：

1. 分析业务系统 live 链路（从业务系统源码出发）——找出 live 一次请求在业务系统内部经过的全部确定性环节，按执行顺序列出每段的：输入
schema、调用的业务系统函数、输出 schema、与上下游段的依赖关系。
2. 把每段构建为可重放的局部链路模拟单元——封装成"给输入 schema → 调业务系统原函数 → 返回输出 schema"的独立函数。这个单元就是 trace 模拟和
attribute tool 共用的最小颗粒。
3. trace 模拟（第一层）= 沿 live 链路按顺序逐段重放——用 trace 节点的 evidence 作为每段输入，重放业务系统函数，对照 trace actual 标
passed/diverged，定位最早分歧。这里"按顺序"是按业务系统链路的执行顺序，不是按 trace 节点排列顺序（虽然两者通常一致，但语义不同）。
4. attribute tool（第二层）= 暴露局部链路模拟单元——把第 2
步构建的局部链路模拟单元中，第一层全量跑不到的部分（相邻函数、参数化查询、配置查询）作为 tool 给 LLM 按需调用。

关键区别

- 之前我说"每个 trace 节点对应一段局部链路"——错的。应该是"业务系统 live 链路的每一段对应一个 trace 节点"——业务系统链路是主，trace
节点是映射到这条链路上的快照。
- 拆解的依据是业务系统源码里的 live 执行结构，不是 verifier 的 trace 形状。trace
节点可能漏记、可能合并多个业务系统环节，但业务系统链路本身是确定不变的。
- 如果 trace 节点和业务系统链路段不是一一对应（比如一个 trace 节点合并了业务系统的两段），拆解时仍按业务系统链路分段，trace
节点作为对照样本时按段对齐。

复用时的真正步骤

把这套方案平移到新项目：
1. 去业务系统源码里分析 live 链路——找到一次 live 请求的全部确定性环节
2. 把链路拆成局部段——每段一个"输入 schema → 函数 → 输出 schema"闭环
3. 每段构建为可重放单元——封装成独立函数
4. 第一层 trace 模拟：沿这条链路按顺序逐段重放，对照 trace actual 定位最早分歧
5. 第二层 attribute tool：把第一层够不到的段/相邻函数做成 tool

不变的东西：拆解对象是业务系统 live 链路、每段是"输入 schema → 函数 → 输出
schema"闭环、第一层全量重放定位最早分歧、第二层按需调用。变的东西：业务系统链路的具体段落划分、每段对应的具体函数。
