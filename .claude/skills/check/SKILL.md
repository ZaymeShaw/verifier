## check agent的构建原则
1. 核心机制不仅是对产生物进行审查，更是对产生机制进行审查（最常见比如运行代码），比如模型生成了一些新的代码时，就有必要触发check agent对新代码进行审查，然后也要包含对历史代码进行审查
2. 保持一致性：所有数据，所有代码，所有组件，都要跟统一，并且统一为最新的口径，如果有落后过时的就全部重跑
3. 最简化原则：所有代码/函数/前端组件等尽量简洁，避免无用/冗余/无效/过时/功能功能，维持奥卡姆剃刀原则，相同功能
4. 协议对齐：协议-项目代码实现-前后端-数据-标准等必须对齐，如果更新就同步更新。包括以下：
+ 一个协议会关联多个项目实现/前端，所以涉及协议的东西尽量少动，如果确实需要改请知会用户确认
+ 相对的，项目实现/前端尽量在协议的范围内更新，发现现有协议无法支持的情况请及时告知
+ 比如说一个项目需要更新，这时候如果它必须更新协议才能继续完成，那这时候不仅它和它对应的协议要改，连它对应协议下的所有项目实现也要对应的改，所以要注意这个关系
+ 所以项目实现/前端实现等，也尽量直接复用协议中出现的代码，尽量做继承和具体类的实现，而不是自己搞一个新结构出来


## check agent做什么
1. 构建当前项目的check agent，用来做代码质量的审查
2. 实测审查当前标准化问题
3. 在代码存在大量更新/用户要求时，检查现有全部代码是否有非标准化的情况，以及根据发生的非标准化的情况检查具体问题，将该测试项目调整回标准化的状态（修改前先向用户确认）

### check agent要解决的问题
check agent的主要职责是对代码进行标准话检查，来避免一些问题，包括但不限于：
a) 过度的规则化：为了优化少数case，将代码通过规则化的方式改的完全仅适配这些case，丢失了应有的泛化能力，在新的case/场景上表现大大降低
b) 局部样本修改：为迎合用户需求，仅修改当前正在处理的数据，导致数据逻辑错乱
c) 只优化展示结果不优化源头逻辑：当你发现某个东西的结果有问题时，不仅要关注怎么把这个结果改对，还要关注这个东西的源头是什么（比如生成他的代码是什么），因为只有源头是对的，你后面重新运行才不会出错，否则你就变成只改结果，下次重新运行还是有问题。也就是你要让生产源头是对的，而且源头可能有多道，你要找到最根本的源头
d) 数据更新不同步不一致：每次修改代码就连接一个新数据/新代码的都不知道哪个是对的，然后前端显示的东西也不更新/仅局部更新。要避免仅新代码仅更新局部数据，要保证前端、后段、代码、数据的一致性，拥有稳定的映射模式
e) 代码/前端组件的冗余/失效/过时：随着项目迭代越来越庞大，更新过程中部分代码/函数/组件开始失效/过时/冗余，前端组件已经无法点击等，并且代码越来越多，难以维护，大量无用功能/代码。你需要：
+ 全局扫描在项目迭代过程中过时、失效、冗余的代码块/功能/前端按钮等，分析失效原因，按照项目的业务意图，向最新意图进行代码的更新，向最新逻辑标准/业务意图进行对齐。对于该类代码常见3种处理策略，请分析方案并向用户确认后执行：1. 过时但仍有业务意义的代码，向最新标准进行对齐 2. 业务上已经完全无用/冗余的代码，确认不被其他组件依赖后进行删减 3. 有复杂依赖的，适当对依赖进行拆解分析，分析最小化调整的方式，对依赖链条进行更新/标准对齐/删减
+ 审核新增内容的必要性，确定其是否会破坏影响现有结构、功能等，避免新增无用功能，同上排查无用/失效/不稳定/依赖混乱代码


重点！仔细思考：对于审查发现的上述问题，给出通用、标准化、不伤害当前效果的（超级重点，要仔细思考解决方案）、合理的、有泛化性的解决方案，在用户确认后执行，告诉用户具体怎么修


### 例行检查事项
check agent需要逐项审查执行：
a) 完整更新：更新代码后，记得重启服务，并且去实测验证，确保更新后符合自己的预想，而不是改完就算了，包括代码api测试、函数测试、后端测试、前端uat测试，mock源头数据后全链路逐项测试（测试链路参考.claude/skills/evals/define/tester.md）
b) 问题可视化与追踪：
+ 把发现的问题放在search-test-case/issue里面，要让用户能看懂是什么问题
+ 将check agent上述的工作内容做成check list，逐一验证是否通过，给出check报告

典型例子：
当前“45岁女性保费10万以上”类case经过多轮修复后，归因已经比较准确。但是一旦切换到“有生存金未领取”这种新的不同的case，归因界面出现形如“大模型归因失败：大模型归因已返回，但没有解决问题：缺少clientAge的期望条件或修改方案；缺少clientSex的期望条件或修改方案；缺少annPremSegNum的期望条件或修改方案”的描述，这里体现了至少3个问题
1. 归因定位结论形如：“L4 LLM输出了合法字段polNoInfo.payAmountDue但因类型是NESTED_MATCH且include值非EXISTS，在query_router.py的_validate_conditions中字段未注册导致被整体清空”。明显没有满足归因的要求，归因的结果没有用，归因不能看成一个简单的分析，归因的定位很接近解决问题了，只是没有去实际改代码，要去把真正的问题找出来，这样输出的归因感觉就没有解决问题。再次说明归因总结的部分要求形如下面参考材料提到的原则1。但之前“45岁女性保费10万以上”这类的归因分析其实已经做的比较好了
2. 明明是生存金相关的归因，但是归因口径的描述出现了clientAge、clientSex、annPremSegNum三个与之不相关的字段，这三个字段是类似“45岁女性保费10万以上”这种分析常见的字段，说明模型当前归因链路已经严重过拟合甚至规则化处理该类问题，完全丧失了泛化能力

##  参考材料
>原则1：归因总结需要给出证据：需要确定各链路的逻辑是正常的，写一些链路测试的脚本，尽量用现有的写好的项目代码，通过import之类的方式引用啥的，比如全链路是l1l2l3l4,中间可能还穿插着一些es/各种函数啥的，那你就做一些l1l2/l1l2l3/l1es之类的类似的这种局部脚本，来看看到底哪里具体有问题。另外，并不是说只有这些链路，我的意思是你要识别一些关键流程和关键函数之类的，看看串到那个点为止链路到底是不是正常的，从而识别出来有没有问题，这个应该是在归因的时候要做的，你归因要去用这个函数进行测试。做这种api测试/局部链路函数/模拟链路测试，看到哪里为止是通的，但是接下来调某个现在链路的函数得到的结果与预期不符，得出结论所以最终问题是来自于哪里
