# 怎么排查
排查用户需求提及关注的代码，分析是否存在通过投机取巧通过测试的情况，包括：
- 通过注入fallback避开业务意义上应该被判定为失败的场景，从而通过测试
- 为了使得测试通过，修改本来正确的标准定义，使得标准不统一/标准错误
- 通过强制构建一些不必要的自定义实现，达成指标
- 权责混乱：通过行为越界（做了超出责任的事）、参数越界（带了不该带的信息）、返回值越界（产了不该产的输出），强行跑通功能，为后续扩展埋下结构性障碍


# 为什么需要做这个验证
测试是为了找出问题，而不是为了通过验证指标。如果为了通过指标，不去解决业务问题而去通过一些投机取巧的办法逃避问题，那就本末倒置了
