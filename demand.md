# 需要拥有的能力

1. 业务和代码理解能力：从api文档中，加载api信息，理解业务背景，解析理解代码逻辑
2. 构建虚拟测试数据，基于业务背景和项目信息，构建或加载输入测试数据/意图
3. 应用执行能力：启动上下文无依赖的独立的环境，构建并启动模拟api服务或者启动/加载现有api服务
3. 批量运行能力：实时或批量运行项目api，获取api输出结果
4. 判断能力：判断api输出是否正确。说明为什么该输出错误
5. 归因能力：当输出错误时，能解析项目代码流转逻辑的情况下，清晰的指出错误的原因，并且为开发提供切实明确有意义的优化方案，归因总结的东西得是有用的，得让开发能知道具体问题点，不要只是模棱两可的说是因为某个模块出问题了，这种描述看了都不知道怎么修复
6. 代码标准/一致性审查：定期检查ai生成代码，保持一致性，对齐项目各处代码/pipeline/agent/数据等的逻辑和标准


# 需要拥有的agent

1. analysis agent：从projects/<project>中(由用户编写基础项目信息，以及对各个agent对补充要求)，读取相关文档信息，抽取项目api、业务背景信息，抽取项目相关信息放到impl/projects文件夹下，包括
    + 理解api调用链路，构建接口文档、接口格式yaml文件等（部分没有完整项目的场景，需要根据application agent生成情况进行调整）
    + 如何mock数据集，构建独立mock方案文档
    + 理解如何对api输出结果进行评判，分析得出评判方案标准文档、评估agent脚本等
    + 理解整个代码运转链路pipeline和关键配置文件、核心函数等，抽取问题归因方案文档、归因agent文档等
2. check agent：对代码进行审查，给用户进行核对，用户确认后，对general_eval项目代码进行更新
3. application agent：根据接口文档，启动业务项目服务
    + 在有现成项目等情况下，启动项目api服务/脚本/pipeline等，形成标准化使用方式
    + 在当前没有现成完整项目的情况下（比如仅有skill），模拟构建agent链路，形成脚本、api服务、pipeline，构建可落地的代码文件，形成标准化使用方式
    + 自测验证标准化使用方式正常可用，将标准化使用方式放在application文件夹中
4. mock agent：结合mock方案文档，构建虚拟测试数据，包括 
    + 模拟构建用户意图
    + 基于用户意图和当前应用上下文，模拟用户下一步输入（可适用单论/多轮交互场景）
5. judge agent：结合评判标准文档，判断一个session中的链路输出是否正确、得出评分等
    + 调用deepseek-v4 pro，启动max思考模式，进行深度分析实时judge（参考env.md）
    + 分解本项目上做好judge需要的工作，在impl/project构建能力强的judge agent
    + 实时调用大模型分析，不应该是秒出结果
    + 输入数据没有提供参考答案时，请同步结合judge分析输出一个参考答案reference
    + 分析尽量生成中文
6. attribute agent：当输出错误时，能结合归因方案文档，从业务项目代码链路中归因出错误原因
    + 请解析
    + 调用deepseek-v4 pro，启动max思考模式，进行深度实时归因（参考env.md）
    + 分解本项目上做好attribute需要的工作，在impl/project构建能力强的attribute agent。
    + 实时调用大模型分析，不应该是秒出结果
    + 分析尽量生成中文

# 实现规范

1. ai编写内容，统一放在impl文件夹下
+ 通用的放在impl下按序构建。最好是构建一些协议类的东西，确定一个规范，包括judge、attribute、前端页面等，然后业务的具体实现按照协议来填这个框就行
+ 项目相关的，放在impl/projects下面的，归属于业务自己的实现，要去实现根据这个协议的具体内容点
+ 部分情况下需要构建一些模版XXX-template.md，放在impl下面，然后用户能够拷贝这些模版，填写模版里面的字段放在/projects/<project>/XXX-template.md，填写好后就能成为 <project>项目的具体标准，ai生成具体的东西落地在/impl/projects/XXX.md，然后具体的实现根据这个标准来进行调整
2. 各个协议之间，必须得能串起来，然后协议的设计要有足够的灵活性，能支持简单的也能支持复杂的s

# 关于协议

> 协议需要能满足这些要求
1. 核心功能区分
2. 关于边界协议怎么写
    - 边界协议需要体现出真正重要的边界区分点
    - 通用性：任何项目上，能能通过这套协议构建边界，而不是有的项目试用这套协议，有的项目不适用
> 模版
    + 目标导向：填写每个模版，都是为了能让用户能通过明确一些东西，填写后就能针对该项目制定出明确的标准、划分好边界等
    + 通用性：任何项目上，能能通过这套模版构建项目对应的标准，而不是有的项目试用这套协议，有的项目不适用
    + 可填写性：模版得是用户能看懂，能填的。这届需要模版有一定的解释模版的成本，同时好填易填，要填的东西尽量不要太多，然后要填的东西应该字段上相对固定但是内容上能相对灵活让用户能好好的说明自己的理解，这个需要针对该模版的目标来调整（现在还是没看懂到底拷贝后用户要填啥）
    + 扮演好的引导：模版要填写的字段，应该要起到好的引导作用，能让用户填写好这些内容后，就能完成实现模版需要的目标，而不是简单的罗列一些东西让用户填，用户不知道咋填然后乱填然后填了也没作用
    + 模版本身提供的字段，就应该要起到一个提供信息量的作用，就是通过你给的要填的字段，用户应该能自然联想到哪些能区分好标准和边界上的东西，然后只要正常不用过多思考的填好，就能逐步让边界区分了，而不是给一个很含糊不知道该填什么好的东西，这样用户可能没有足够能力给出好的输入，模版也就失去了他的作用

        + 获取live请求

## 怎么做judge

> 重新理解输入背后的核心用户意图。评估业务服务输出是否cover用户需求，根据不同需求会分两种可能（关于这点，其实我想听下你的理解看怎么做比较好，我也没想好应该怎么做）
1. 纯粹评估有没有完成好用户意图，不管实际条件如何（包括可能的上下文局限、系统局限等因素）
2. 评估在当前系统条件下，完成了系统可做好范围内的多少（比如说在系统当前数据库配置等局限下，最多只能做到这样，即便不是完美完成用户意图，也属于一百分
3. 我觉得需要协议让用户填一个东西。分清楚项目在上述两项上面的一个边界点，协议应该给一些东西来让用户写，来确认这个边界，明确协议尽量简单。通过模版的方式来做吧，做一个judge_boundary-template.md放在impl里面让用户拷贝填写模版
    + 边界协议不需要填写所有judge需要的东西，只需要填写针对项目评估边界上面的东西，所以协议要填写的东西应该也只需要让用户围绕这方面填最小化的在各项目上可能变化的点
    + 边界是用来完善和进一步确定judge构建逻辑，请结合此对项目的judge进行优化，确保项目judge满足边界协议的要求，以及确保当前协议能满足judge的需要，必要的时候也要调整judge协议（不是说边界协议，是judge协议）




## 怎么做归因（你考虑如何优化协议）

> 归因要做到得东西
1. 归因分析，必须以能否解决问题为导向：
    + 如果没有发现真正的问题，你应该要将其视为没有成功，然后继续优化。给出明确的修改方案，定位到具体的点（但是不直接改代码）
    + 有可追溯、可执行、可直接使用、采取后能直接解决问题的证据链路，确定链路中哪些正常哪些不正常
    + 归因总结的东西得是有用的，得让开发能知道具体问题点点，不要只是模棱两可的说是因为某个模块出问题了，我看了这种描述都不知道怎么修复
2. 将业务项目构建为trace，拆解进行分析
    + 将api链路拆解为可操作的trace，包括大的流程，以及小的一些关键函数组件，都记录在trace中，执行局部链路模拟测试脚本，验证根源问题发生点
    + 模拟trace分析尽量用现有的写好的项目代码，通过import之类的方式引用啥的，避免编写一些不存在的东西出来，最后得到的结论也与本项目无关
    + 看看在整个trace中，串到哪个点为止链路到底是不是正常的，从而识别出来有没有问题。做这种api测试/局部链路函数/模拟链路测试，看到哪里为止是通的，通过比如发现trace链路的某个节点上得到的结果与预期不符，得出结论所以最终问题是来自于哪里
3. 归因问题时，要体现错误代码位置，指出具体哪个大模块中的哪个组件/函数与预期不符，预期是怎么样的实际时怎么样的。原因是哪个配置/prompt/代码链路的误差导致。后续该怎么优化
> 归因要避免的问题
+ 映射混乱：分不清正确哟字段/配置/枚举值正确映射关系，胡乱对应
。。。

# 通过系统边界确定评估范围

应在协议中体现，分析系统是怎么样的来在协议范围内制定judge/attribute的impl或impl/project实现，而非每次由judge独立判断（注意这点）

## 什么是评估范围

简单来说就是，在我们评估同一个事情时，在不同的约束下，得出的结果也是不一样的
+ 在完全理想的情况，用户也没有给出具体约束时，我们以最理想的范围内评估，通过看回答能否满足用户意图为核心评判因素，能满足多少就是多少分
    - 典型场景：用户直接提供输入、模型输出。此时用户没有给出任何约束项，因此应以输出能否满足，满足多少用户意图为唯一判断标准
+ 在用户给出了系统的情况下，那我们应该在系统处理责任范围内进行评估
    - 比如回答不完全满足用户意图，但是不满足意图的卡点是因为系统之外的因素没做好导致，那这种就不是我们责任范围内，因此不应该以此扣分。评估系统仅评估系统范围内能做到多少分
    - 例子1：客户搜索下游数据库缺乏某关键字段导致无法搜索特定类型客户，这种情况不是客户搜索代码系统能优化的，因为用户数据存储不是客户搜索系统的责任范围，此时该问题不应该扣客户搜索结果的评分
    - 例子2：由于prompt编写错误、配置文件配的有问题、pipeline构建有问题、模型能力不够等因素，导致客户搜索接口没有产出正确结果，此时问题就是客户搜索的，应该扣他的分

## 为什么需要评估边界

有时候我们需要评估理想情况下应该做的怎么样，有时候我们想评估我们责任范围内做的怎么样，
application agent根据系统的情况，来评判我们的责任范围，划定judge agent的评估边界


## 我们要评估什么

简单来说，就是在我们责任范围内，看我们的系统做的怎么样，无关乎具体实现
judge评估我们责任内，系统的输出对用户的意图完成的怎么样
attribute评估我们责任内，系统出问题的情况下分析全链路看具体是哪里有问题
