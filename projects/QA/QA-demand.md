# QA 测评需求

- 业务目标：对给定输入、实际输出、参考答案和上下文进行样本级质量判断与多维评分。
- 范围：上传输出评估、单轮 QA、scenario 区分、judge、可选手动归因和聚类。
- 非目标：不调用或替代被测 QA 生产系统，不把 client_search 等项目特例写入公共协议。
- 核心场景：golden answer 正确性、context 忠实性、弱参考质量三类场景。

## 原始业务说明

关于QA场景分析
你参考projects/QA/material/qa-evaluation-system-design.md的需求
这个场景我希望能直接给输入、输出和参考的golden answer。常见情况是我提供形如projects/QA/material/qa-evaluation-system-design.md中的数据集，你解析出用户输入，输出（对应api输出），然后模型通过参考输出和用户意图，判断实际数据对不对，给一个评分之类的东西，最好是多维度评分
你可以这样，比如我上传了这个数据集包含这三个字段，然后你将他拆出来放到前端协议对应的地方（你看看这样会不会有什么问题）

然后这个的话对前端协议有一些新的更新要求，比如运行输入这个地方，project这个地方应该得是个可枚举的下拉框，然后选不同东西就对应不同项目，现在应该是有QA和client_search这两个项目  





分歧点和坑点：
+ 文件上传：依然统一json格式，做一些格式转换的工具到时候提前方便转下就好
+ 前端 project 要改成枚举下拉：live和归因总结都需要这样改，对可以整个统一的api
+ 协议本来就应该是相对通用的而不是仅适配一个项目的（比如仅适配client_search）,如果协议不合适不通用就要适当的改协议了（针对多维评分、归因和 cluster 也要按 QA 错误类型体系对齐）。我再强调下，协议做通用的，然后具体项目实现上可以在协议基础上针对项目特性有些许不同，这个之前说了，现在也是按这个框架做的
+ QA 不应该调用被测 QA 系统：这个可以忽略，只是开发者对另一个系统的要求，我们按照它的待评估内容看在我们这个系统上怎么做
+ 逐条评估模式不能按文件统一：没看懂这是啥坑
+ RunTrace 语义会变，在这个项目上确实，可能得调整成输入是说的3个字段，然后输出从输入中直接读取，judge再独立判断这样（我理解这个好像是通过application agent来做）
+ 关于逐条评估模式的坑点：我感觉这种情况，最好是不归为同一个场景，但是我不确定怎么区分比较好，你有什么建议？
+ contexts 应该算输入的一部分，你看看是不是要分，应该是场景不同导致的，你看看是不是要从同一个QA里面，分出多个略有区分性的场景，然后字段也稍有不同这样
+ 字段映射页没必要，这是他自己再别的项目的需求
+ 归因和 cluster 要不要也升维、多维评分适配
+ Project 下拉会影响所有页面状态：所以我说了要统一协议，保持一套逻辑，这样切换的影响也没那么大
+ QA 项目不该污染 client_search 的页面体验：对，看下怎么合理的做一套更通用的，不要抢行硬加，说了不同项目在协议的基础上可以调整
+ 输出对应 api 输出”这个概念要小心：对，换个好的名字
+ check agent 要新增 QA 专项检查：你真对impl/QA里面的check agent可以做这种调整
+ 第一版边界建议：还是那句话，你看下怎么区分边界协议之类的
+ 我觉得scenario可以作为一个项目特定的实现，可以协议允许各个项目有一些特定的选择，然后QA项目选择了scenario这个扩展
