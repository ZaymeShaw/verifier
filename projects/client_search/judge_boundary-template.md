# Judge Boundary - client_search
限制：有些用户受限于当前上下游依赖的数据库中没有相应字段或枚举值的，可能无法完全满足用户意图搜索出正确字段，此时应该按照业务系统能力边界判定是否正确



- 因为上下游的es数据库能力不是由我们系统能力强弱决定的，我们只能做到在我们能决定的范围内做到最好，所以评估只评价我们当前系统做的怎么样了，而不评估由于上线限制导致的偏差（指我们无法控制的es数据库这种），这种问题我们无法衡量和解决
- 但如果是我们可以优化的部分（比如大模型、项目配置prompt.md/各种config文件、项目代码、后处理逻辑、pipeline、模型能力本身等）做的不好，那就应该纳入衡量范围里面
- 由于prompt、项目配置、后处理逻辑、pipeline等模型等系统内部内容都有可能出错，所以他们不能成为绝对标准
- 只有系统外部的约束才是硬标准（比如下游es依赖），下游数据等东西

- judge的核心是：看api结果输出，从下游配置文件/枚举值文件中了解，api输出的搜索字段与客户输入的搜索意图是否一致。具体点字段其实不完全是重点，因为标准答案可能不止一个，核心是业务语义正确，在规定的枚举值内，根据api结果能搜出正确客户列表即可
    - 数据库是es，这是你的核心判断依据，根据es的表结构、查询规则、里面字段的枚举值来进行判断（数据库信息，请你从原项目的配置/枚举值文件中获取）
- 我们最终需要的是能从数据库中搜索到正确客户的parser，因此如果judge发现数据库搜索结果等价的情况下，即便和prompt.md/config.md/后处理逻辑/pipeline的要求稍有不同也是可以接受的
    - 什么情况下会出现等价查询，这取决于用户搜索语义和es查询语法（当下游不可用时，必须要做好这点的判断）
    - 如果无法判断的话，可以尝试看能不能做一下模拟查询，看查出来的数据是不是用户意图想要的，从而让judge能评估系统输出的正确性/评分
