## 一、各项目判断链路（说明为什么需要/不需要某些信息量）

### marketting-planning-intent（标签比对）

| 判断步骤 | 说明 | 需要的信息 |

| 1. 读 query | 理解用户说什么 | 原始输入 |
| 2. 读 intent_label（API 返回） | 评估对象 | extracted_output |
| 3. 读参考 intent | 判定锚点，比对的靶子 | expected_intent |

## 三、结论：当前上下文工程是否合理、是否有确实的缺失

### 合理的情况

1. **mpi 的 evaluation 文档只有 2 句话（316 chars）——够用。** 它的判断只需要一句话就能定义清楚（"判断 intent 标签是否匹配参考"），不需要像 client_search 那样写 54 行。多写对判断精度无增益，还占 context——这符合奥卡姆剃刀。

2. **mpi 的 capability_manifest 为 `{}`、semantic_equivalence_rules 为 `{}`、judge_standard 为空字符串——合理。** 正如判断链路所示，它们在 mpi 的判断链中不扮演任何角色。注入内容反而是浪费 context。

3. **mpi 的 reference_contract 为 `{}`——合理。** mpi 的参考意图（expected_intent）通过独立字段传入，不是通过 reference_contract 结构。空字典只是没有额外契约，不影响判断。

4. **client_search 的 context 比其他项目多 10 倍——合理。** 它的判断链路需要判断字段、操作符、值、逻辑关系、等价变换、w/m/e 汇总、下游结果集验证，自然需要更丰富的信息。不是项目偏心，是业务复杂度决定了这些。

### 确实的缺失

| 问题 | 判断链路影响 | prompt 原文对比 | 修复方向 |
|------|------------|---------------|----------|
| **mpi: expected_intent = `null`** | 第 3 步（参考锚点）断裂。judge 在没有参考标签的情况下，无法做标签比对；实际退化为"LLM 猜这个 query 应该是哪个 intent"。 | mpi prompt: `"expected_intent": null` vs 判断需要: 显式的标签值 | **数据侧**: case 数据里补 expected_intent |
| **cs: project_type = `""`（空字符串）** | 第 0 步（告诉 judge 这是什么项目）不起作用。但 client 有大量项目专属信息（evaluation/judge_standard/judge_boundary 三个大文档）补偿了这个空值，实际影响比看起来小。 | cs prompt: `"project_type": ""` vs 其他项目: `"single_turn_marketing_intent_recognition"` | **context engineering**: adapter 注入 project_type |

**以上就是全部。** 其他所有"覆盖率不足 100%"的情况都是"该项目不需要这条信息"的合理状态，而不是缺失。
