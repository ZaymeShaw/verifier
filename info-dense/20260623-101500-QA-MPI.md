## 一、目标分解与系统分解

### QA 项目

**业务目标**：对已产出的 QA 问答（question + actual_answer），在有/无 reference/contexts 的情况下，评估答案质量并归因错误类型。

**业务系统链路**：
```
question + actual_answer (+ contexts + golden_answer)
  → scenario 推断 (gold/faithfulness/weak)
  → 按场景选评估维度 → 逐维评分 → 汇总 verdict
  → 归因错误类型 (error_taxonomy)
```

**测评系统介入点**：
- adapter 接收上传数据 → 归一化 → trace 构建
- judge 按 scenario 分类评估
- attribute 按 error_taxonomy 分类归因

### marketting-planning-intent 项目

**业务目标**：评估营销规划意图识别系统（intent-recognition 单轮接口）的标签、置信度、槽位的正确性。

**业务系统链路**：
```
user_query → intent-recognition API → intent_label + confidence + slots
  → judge 比对 expected_intent 标签
  → attribute 定位标签/置信度错误原因
```

**测评系统介入点**：
- adapter 构建请求 → 调用 intent-recognition API
- judge 标签比对（契约式）
- attribute 从 exec_trace 定位失败节点

## 三、结论

### QA：当前上下文工程整体合理，但有一个关键缺失

**合理项**：
1. scenario 推断、score_dimension、error_taxonomy、application_boundary 均 100% 覆盖且有效，满足 QA 判断链路
2. golden_answer exact match probe 高效处理了最常见场景
3. fallback 机制（local_evidence_probe + sample_label）完整，确保 LLM judge 失败时有兜底

**关键缺失**：
- **LLM judge 失败原因不可追踪**（信息损失率 70%）：qa-context-supported case 返回 uncertain，但 adapter 无法获取 LLM judge 为什么返回 uncertain。归因时只能写 "judge LLM 的原始 prompt 与 raw response 不在 source_file_catalog 中"。

### marketting-planning-intent：数据侧和 prompt 注入都有缺失

**合理项**：
1. query/intent_label/application_boundary 覆盖完整
2. evaluation 文档恰到好处（2 句话定义清楚，符合奥卡姆剃刀）
3. reference intent / confidence / fallback 等在数据侧部分覆盖

**关键缺失**：
1. **critical_intent_dimensions 未注入（信息损失率 24%）**：adapter 的 build_judge_context 已定义 5 个维度，但未注入 judge prompt（cs 有相同的字段且已注入）。这是一个 context engineering 层面的修复——adapter.build_judge_context() 已定义了字段但 build_intent_frame() 也传了一份，实际注入 judge prompt 时这个字段丢失了
2. **expected_intent 数据缺失（信息损失率 47.5%）**：9 条 mock 中部分 case 的 reference.intent 为空。这是数据侧问题，需要补全 expected_intent
3. **attribute 源码检索不完全（信息损失率 20%）**：STAGE_FILE_PREFIXES 限定了 8 个文件，但外部意图识别系统的 prompt 文件（intent_recognition 的 LLM prompt 和正则规则）不在 catalog 中
