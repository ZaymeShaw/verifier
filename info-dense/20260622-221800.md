# 系统信息量评估 — judge agent

> 生成时间：2026-06-22T22:18+08:00
> 评估方式：逐项目分析 judge 要做哪些业务判断 → 推导每个判断要什么信息 → 看当前 prompt 给了什么

---

## 一、各项目 judge 的判断逻辑及信息依赖

### marketting-planning-intent（简）

**judge 实际上要做什么：**
读 API 返回的 intent 标签，跟 case 的预期 intent 标签做比对，判断相等还是不相等。

**业务判断链路：**
```
query → API → intent_label
                   ↓
预期 intent_label  ← judge → 判定 equal? → 输出 verdict
```

**贯穿这条链需要的上下文：**
| 按顺序 | 作用 | 必要性论证 |
|--------|------|-----------|
| 本次 query `{query:"我想买车险"}` | 理解用户说了什么 | 没有它 judge 不知道在评什么 |
| API 返回的 intent_label | 要评估的实际输出 | 没有它 judge 没有评估对象 |
| 参考 intent_label | judge 的判定锚点 | 没有它 judge 只能靠 LLM 猜"这个 query 该是什么意图" |
| application_boundary | 告诉 judge 边界是意图识别，不涉及后续规划 | 没有它 judge 可能把标签比对延伸成流程评估 |
| judge_boundary | application_boundary 的内容文档 | 同上 |
| project_type | 告诉 judge 这是意图识别项目 | 没有它 system prompt 不知道调取哪套通用规则 |

**不需要的：** capability_manifest（标签比对不需要操作符体系）、semantic_equivalence_rules（标签是枚举，不存在同义表达问题）、condition_comparison（不需要 w/m/e 框架）、critical_intent_dimensions（只有一项比对，无需多维度分解）、field_provider（没有字段体系）、judge_standard（evaluation + boundary 已盖全），**expected_intent**（在数据层面实际是空的"null"，但对 mpi 这是核心锚点——这不是"不需要"，是业务上需要但数据没给）

### client_search（繁）

**judge 实际上要做什么：**
读 query → 理解用户想搜什么客户 → 读 parser output（字段、操作符、值、逻辑关系）→ 逐字段判断是否正确 → 用 w/m/e 框架汇总偏差 → 有下游结果集时做最终验证。

**业务判断链路：**
```
query  "25岁女性客户"
  ↓ 理解
用户想搜"年龄=25 且 性别=女"的客户
  ↓ 比对 parser output ↓
{clientAge:RANGE:25~25, clientSex:MATCH:女} → 操作符对吗？值对吗？缺吗？多吗？
  ↓
w/m/e 汇总 + 下游 ES 验证
  ↓
verdict
```

**贯穿这条链需要的上下文：**
| 按顺序 | 作用 | 必要性论证 |
|--------|------|-----------|
| 本次 query | 推理意图起点 | 无此链断裂 |
| extracted_output (parser output) | 评估对象 | 无此无判断目标 |
| evaluation 文档 | 告诉 judge 核心判断原则（错/缺/多分层、ES 证据优先、不继承历史） | 没有它 judge 不知道当前项目用什么标准 |
| judge_boundary 文档 | 边界（什么归系统、什么归 ES） | 没有它误判 ES 缺字段为系统错误 |
| judge_standard | 具体判定步骤（先读 boundary → 逐字段 check → w/m/e） | 没有它 LLM 按通用规则写 expectation，可能漏分层 |
| capability_manifest | 每个字段允许的操作符和值类型，判定"操作符用对了没" | 缺失则 judge 靠先验猜 GTE/LTE 是否合法，等价于不做操作符合规检查 |
| critical_intent_dimensions | 多维意图分解框架 | 缺失则单条 LLM 拆 expectation 不稳定 |
| semantic_equivalence_rules | 等价变换知识（RANGE 25-25 ≈ GTE 25 LTE 25） | 缺失则错判等价条件为 wrong |
| condition_comparison | w/m/e 对比结果 | 缺失则 LLM 自行做 w/m/e 判定的框架，不稳定 |
| application_boundary | 是否可用下游结果集 | 没有它 judge 可能声称结果集验证过但实际上没做 |
| reference_contract | 无参考标准时的回退 | 二线信息，缺了不影响核心判断但影响极致准确 |

**不需要的：** 无。上面每一条对 client_search 的完整判断链路都是必要的。

### QA

**judge 实际上要做什么：**
根据 scenario 不同，读 question + actual_answer + reference/context → 按场景维度评分 → 输出 verdict + score_details。

**业务判断链路：**
```
question + actual_answer (+ reference.golden_answer / contexts)
  ↓
确定 scenario（gold/faithfulness/weak）
  ↓
按场景选评估维度 → 逐维评分 → 汇总
```

**贯穿这条链需要的上下文：**
| 按顺序 | 必要性论证 |
|--------|-----------|
| question + actual_answer + golden_answer/contexts | 没有则没有评估对象 |
| evaluation 文档+score_dimensions | 告诉 judge 每个场景的评估维度和 metric 口径 |
| judge_boundary | 限制 judge 只评估 answer 质量不评估 QA 服务的实现 |
| error_taxonomy | 收敛归因错误类型 |
| application_boundary | 告诉 judge 不需要外部服务 |

**不需要的：** capability_manifest（QA 不涉及字段操作符体系）、semantic_equivalence_rules（answer 质量不存在"同义等价"的概念）、condition_comparison（不需要 w/m/e 覆盖框架）、judge_standard（evaluation 已定义各场景的要求）

### marketting-planning（中）

**judge 实际上要做什么：**
读契约（expected_stage + required_path_types + required_cards + allow_fallback）→ 读 output（stage + card_summary + event_summary + fallback）→ 逐项契约比对 → 差异汇总。

**业务判断链路：**
```
契约 {stage:planning, path:[premium_growth], cards:..., allow_fallback:false}
  ↓
output {stage:planning, card_summary:[...], event_summary:{completed:true}, fallback:{used:false}}
  ↓
逐项比对 stage = ?, path ⊇ ?, events ⊇ ?, fallback < allow？
  ↓
差异汇总
```

**贯穿这条链需要的上下文：**
| 按顺序 | 必要性论证 |
|--------|-----------|
| query + turns | 意图起点 |
| extracted_output | 评估对象 |
| reference_contract（契约） | 没有契约，mp 的 judge 只能靠 LLM 猜"预期是什么" |
| evaluation 文档 | 定义了契约式评估方式 |
| judge_boundary | 多轮+SSE 边界 |
| critical_intent_dimensions | 多维度判断分解 |
| application_boundary | 外部依赖状态 |
| stage_rules | adapter 提供的澄清/规划/兜底规则说明 |

**不需要的：** capability_manifest（状态机校验不需要操作符体系）、semantic_equivalence_rules（契约比对不存在同义问题）、condition_comparison（契约式比对不需要 w/m/e 框架）、field_provider（无字段体系）

## 三、结论

### 真正的缺失（损害判断链的）

| 问题 | 影响程度 | 说明 |
|------|---------|------|
| **mpi: expected_intent 为空 `null`** | 高 | mpi 的判断链路是标签比对，但参考标签在 prompt 里是 `null`。judge 在没有锚点的情况下只能用 LLM 先验猜"这个 query 应该是什么意图"，等于放弃了"比对"这个核心机制，退化成了"凭感觉判断"。这不是 context engineering 的问题，是数据侧没有配。 |
| **cs: project_type 为空 `""`** | 中 | system prompt 中缺少 project_type 标识，但 system prompt 里有大量的项目专属规则补偿。client_search 的 project_type 在不使用 adapter build_intent_frame 时没有被正确注入。 |
| **QA: 无 field_provider** | 低 | QA 的 error_taxonomy 通过 adapter 硬编码注入（`error_taxonomy: ["hallucination","over_refusal"...]`），不走 field_provider 按需检索。这只限制了"如果 taxonomy 很大时不能按需查"，当前的硬编码方案够用。 |

### 本评估过程暴露的方法问题

1. **capability_manifest 对 mpi/QA/mp 覆盖率 25% 不是损失。** 因为它们根本不需要。上一份报告把"覆盖率低"等价于"信息损失"，这是错的——信息损失只在"需要但没给"时才成立。
2. **e10 (expectation 原子化)不是缺失。** 各项目的 evaluation 文档已经明确定义了 judge 怎么产 expectation（mpi:一条比对; QA:scenario 维度; mp:契约式比对; cs:w/m/e 框架）。只是定义方式不是原子化的 JSON list，而是叙述式——这究竟是"缺失"还是"意图清晰"要看具体项目。
3. **之前报告里说"mpi/QA/mp 损失 40%"，这个数字没有意义。** 因为那些信息量它们本来就不需要。
