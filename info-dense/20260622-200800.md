## 结论：上下文工程是否合理、是否该填补或去掉

**应该填补（上下文缺了，导致 judge 做不出正确判断）：**

1. **capability_manifest → 对其他3个项目缺，必须补。** 没有字段定义（operators/value_types），judge 无法判断"这个字段的操作符用对了没"——这不是优化问题，是判断框架缺了基座。judge 现在靠 LLM 先验猜，错了都不知道。

2. **semantic_equivalence_rules → 对其他3个项目缺，必须补。** 没有等价规则，"25岁" vs "年龄=25" 这种同义表达会被判 not_fulfilled，假阴性高。这个在 client_search 的条件对比 tool 里已经解决了，但其他项目不受益。

3. **judge_standard → 对其他3个项目缺，必须补。** mpi/QA/mp 的 judge_standard 是空字符串。这意味着这些项目的 judge 只凭 system prompt 里的通用协议做判断，没有任何项目专属的判断步骤、边界优先级、判定流程——输出质量完全取决于 LLM 当天的状态。

4. **condition_comparison → 对其他有"覆盖类评估"需求的项目缺，必须补。** w/m/e 判定必须有"预期是什么、实际是什么"的对比框架。client_search 的 condition_compare tool 就是干这个的，但其他项目没有等价物。

**应该去掉（上下文有冗余或可压缩的）：**

1. **evaluation 文档 → 可以考虑压缩。** client_search 的 evaluation 文档（5469 chars）有大量和 judge_boundary + judge_standard 重复的内容（边界原则重复了两遍，readme/prompt/config 引用多处交叉）。实际上 evaluation 只需要"本条 case 的评估目标"，而边界和流程应该只由 judge_boundary + judge_standard 承载。mpi 的 evaluation 只用了 316 chars 说了一句话，就够用了——但另一方向又太简略了没有提供任何业务背景。

2. **protocol_tool_results → 可以考虑按需而非常驻。** client_search 的 protocol_tool_results 在 prompt 里占了约 1500 chars 的完整 JSON，但仔细对比，condition_comparison 字段提取出来的内容（wrong/missing/extra 判定）已经覆盖了协议工具返回的核心信息。protocol_tool_results 里的完整原始证据（四段 evidence 数组）对 judge 来说很少用到——这个可以考虑移到 tool 按需拉取而不是同步注入。
