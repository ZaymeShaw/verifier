## 二、结论：评估当前上下文工程是否合理，是否应该填补某上下文或者去掉冗余上下文，从业务角度说明

### 真正的缺失（损害判断链路的）

| 问题 | 影响项目 | 信息损失率 | 业务角度理由 |
|------|---------|-----------|------------|
| **QA 无 source_file_catalog** | QA | 100（该项完全不可用） | QA 的 source_file_catalog 为空列表，attribute agent 没有任何文件可查。但 QA 的业务场景是答案质量评估（不涉及外部服务），错误类型是语义性的（hallucination/over_refusal/contradiction 等），不需要源码级归因。所以**这不是业务意义上的缺失**——QA 的归因判断链路不需要源码。 |
| **mpi expected_intent 为 null（从 judge 继承）** | mpi | 47.5 | judge 侧的 expected_intent=null 导致 compact_judge 中没有参考标签锚点，attribute 在重建 expected behavior 时只能靠 LLM 先验猜。这是数据侧问题，不是 context engineering 问题。 |
| **cs attribute 无 condition_comparison** | cs | 75 | compact_judge 中 w/m/e summary 存在，但原始 condition_comparison 的详细字段级对比结果不传给 attribute。attribute 无法直接看到"哪个字段 expected 什么 actual 什么"，只能从 compact_judge 的 business_expectations 和 fulfillment_assessments 间接推断。但 compact_judge 截断后（2500chars/条）可能丢失细节。**建议：compact_judge 保留 condition_comparison 结构，或至少在 project_attribute_context 中传入精简版 condition_comparison**。 |
| **cs attribute 无 value_mappings/enhanced_rules** | cs | 70/60 | attribute 在归因 parser 步骤的 value_map / routing_pattern_match 时，不知道项目配置的映射规则。但这可以通过 search_source_file 按需读取 YAML 获取——不是信息缺失，是**获取路径存在但 agent 需要自己主动查**。system prompt 中已说明"按需读取源码文件"，所以这不是上下文工程的缺失。 |

### 当前上下文工程合理的部分

1. **通用 system prompt（6 步归因流程）**：含核心流程、输出格式硬性要求、关键规则，所有项目共用。对 cs/mpi/mp 是足够的，对 QA 略微偏重（QA 不需要源码级分析），但 QA 项目属性决定它不会触发 search_source_file 调用，所以不影响。

2. **attribution_spec 的差异设计**：cs 30 行（9 步推理链），mpi 15 行（6 阶段 + 支持证据要求），QA 21 行（15 类 taxonomy + 输出要求），mp 17 行（9 阶段 + 判断原则）。长度与业务复杂度正相关，符合奥卡姆剃刀。

3. **compact_trace + compact_judge 的通用截断策略**：所有项目共享同一套 _compact_trace/_compact_judge，各自项目独有的字段由各自的 build_attribute_context 补充。大字段（extracted_output）10000chars 截断，中等字段（business_expectations）2500chars 截断。当前预算内合理。

4. **cs 的 attribute_quality_gate**：cs 是唯一有质量门的项目（4 项目中最复杂），attribute_quality_gate 定义最小证据要求和预期输出。这是合理的——cs 的归因链条最长，需要质量门兜底。

5. **mpi/mp 的 earliest_stage_order + ext_repo 源码按阶段筛选**：根据 execution_trace 中哪些阶段失败来筛选 ext_repo 文件（上限 8 个）+ 指定阶段遍历顺序。这是精准的上下文工程——只加载与失败阶段相关的源码，节省大量 token。

6. **judge 侧工具不传给 attribute**：condition_compare、value_mappings、enhanced_rules、capability_manifest、semantic_equivalence_rules 都在 judge prompt 中用于判断，但 attribute 可以通过 compact_judge 的 summary 获取判断结果，必要时通过 search_source_file 读取源码。**这是合理的设计——attribute 不重复 judge 的判断工作，而是聚焦于源码级根因定位。**

### 可优化的方向

| 优化项 | 影响项目 | 说明 |
|--------|---------|------|
| **compact_judge 中保留 condition_comparison 简版** | cs | 当前 compact_judge 包含 wrong/missing/extra 列表和 condition_assessments，但没有完整的 condition_comparison 结构。如果在 compact_judge 中加入一个精简版的 comparison_summary（expected_conditions vs actual_conditions 的字段级对比），attribute 不需要额外 tool call 就可知晓具体偏差。但影响较小——必要性中等，不建议优先处理。 |
| **去掉 adapter 中 source_config_paths 的冗余注入** | cs | source_config_paths 与 source_file_catalog 信息重叠。catalog 已有 key→path→size→description 的完整映射，source_config_paths 只是重复了部分 path。当前 context 影响不大（4 个路径），可优化但不是问题。 |
