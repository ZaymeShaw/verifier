# 归因 tool工具引用不合理导致归因质量差

---

### 💬 此般浅薄 · 2026-06-24 16:10

> **标记**：`提出者` · `验证者` · `涉及 agent: attribute`

关于工具调用的优化。

我建议你的工具调用，直接引用业务系统原函数，不只是去看业务系统 prompt 这种。
你的归因tool目标是分析出从调用链路哪个地方开始有问题：
+ 你要去获取原业务系统的代码，抽取原业务系统的关键函数，并将原业务系统函数抽取为项目级tool工具
+ 然后通过使用这些tool等方式，从而对整个业务系统trace进行模拟调用，进而归因整体调用链路最初问题发生时间
+ 将不符合预期的模拟调用抽取出来，复盘出再原业务系统中最开始不符合预期的问题时间点/问题使用函数/问题模块是哪个
我希望我能从attr的结果中，看出代码链路具体是哪里、哪个函数、哪个模块开始有问题，以及这样说的证据
- 需要通过模拟原项目函数调用的方式确认这点，你需要构建能实现这些东西的tool，可能要实现项目级的tool放在impl/projects/<project>/tools中进行合理使用
- 不要将归因看成一个简单的分析，归因的定位很接近解决问题，只是没有去实际改代码，但是仍然要去把真正的问题找出来（解决的是业务系统而非测评系统的问题），要做的事情接近于做问题修复前的全盘分析
- 要观察最新的report.md报告，从中提取对应的attr summary结果，并分析是否满足归因的需求
