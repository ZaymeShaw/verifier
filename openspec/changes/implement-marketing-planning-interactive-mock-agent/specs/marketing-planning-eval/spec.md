## MODIFIED Requirements

### Requirement: Marketing-planning evaluation uses unified pipeline with project-owned boundaries
The system SHALL evaluate `marketting-planning` through the existing unified verifier pipeline while supporting the extensible case protocol for single-run, static-turn, and interactive intent-driven cases. Marketing-specific request normalization, session isolation, SSE extraction, reference contracts, interaction policy, and boundary construction SHALL remain inside the project adapter/docs.

#### Scenario: Multi-turn intent runs as interaction when declared
- **WHEN** a marketing-planning mock or provided case declares `interaction.mode = interactive_intent`
- **THEN** the verifier evaluates it as a bounded conversation loop using the project adapter interaction policy and the existing unified pipeline outputs

#### Scenario: Single-run marketing-planning cases remain valid
- **WHEN** a marketing-planning case has no `interaction` envelope or declares `interaction.mode = single_run`
- **THEN** the verifier continues evaluating it through the existing single `run_chain` path

#### Scenario: Path types are execution intent
- **WHEN** a case requires or forbids `path_types`
- **THEN** the adapter treats them as execution-graph intent/reference evidence and judge normalization checks missing or conflicting path choices before accepting a verdict

#### Scenario: Mock case runs through unified chain
- **WHEN** a `marketting-planning` mock case is submitted to `/api/run_chain` or batch APIs
- **THEN** the result contains RunTrace, judge, attribute, cluster/check where applicable, and frontend view data produced through the generic pipeline without project-private verifier endpoints

#### Scenario: Boundary is produced before judge
- **WHEN** judge runs for a `marketting-planning` trace or interactive turn trace
- **THEN** `RunTrace.project_fields.application_boundary` already contains current-case boundary information generated by the adapter

#### Scenario: No project-private verifier endpoint is added
- **WHEN** marketing-planning interactive demand updates are implemented
- **THEN** the system reuses generic verifier APIs and does not add a marketing-planning-only backend route

#### Scenario: Primary stream path avoids split-brain
- **WHEN** live marketing-planning execution is configured or inspected
- **THEN** the verifier uses the primary `/api/v1/marketing-planning/stream` path as the business path and does not split evaluation semantics across intent-recognition, execution, chat, or other internal endpoints
