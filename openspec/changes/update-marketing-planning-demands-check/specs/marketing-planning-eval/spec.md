## MODIFIED Requirements

### Requirement: Marketing-planning evaluation uses unified pipeline with project-owned boundaries
The system SHALL evaluate `marketting-planning` through the existing unified verifier pipeline while keeping marketing-specific request normalization, session isolation, SSE extraction, reference contracts, and boundary construction inside the project adapter/docs.

#### Scenario: Multi-turn intent remains isolated per case
- **WHEN** marketing-planning mock or provided cases describe a user intent across multiple turns
- **THEN** request normalization preserves the intent/turn/session contract and isolates session state by case unless the case explicitly requests a shared session

#### Scenario: Path types are execution intent
- **WHEN** a case requires or forbids `path_types`
- **THEN** the adapter treats them as execution-graph intent/reference evidence and judge normalization checks missing or conflicting path choices before accepting a verdict

#### Scenario: Mock case runs through unified chain
- **WHEN** a `marketting-planning` mock case is submitted to `/api/run_chain` with mock mode enabled
- **THEN** the result contains RunTrace, judge, attribute, cluster, check, and frontend view data produced through the generic pipeline

#### Scenario: Boundary is produced before judge
- **WHEN** judge runs for a `marketting-planning` trace
- **THEN** `RunTrace.project_fields.application_boundary` already contains current-case boundary information generated by the adapter

#### Scenario: No project-private verifier endpoint is added
- **WHEN** marketing-planning demand updates are implemented
- **THEN** the system reuses generic verifier APIs and does not add a marketing-planning-only backend route

#### Scenario: Primary stream path avoids split-brain
- **WHEN** live marketing-planning execution is configured or inspected
- **THEN** the verifier uses the primary `/api/v1/marketing-planning/stream` path as the business path and does not split evaluation semantics across intent-recognition, execution, chat, or other internal endpoints

### Requirement: Marketing-planning output and reference remain compact and comparable
The system SHALL represent marketing-planning output as compact stage/event/card/session/fallback/error summaries and SHALL represent references as comparable structured contracts rather than exact free-form text.

#### Scenario: SSE or provided raw output is extracted
- **WHEN** raw marketing-planning SSE/provided output is processed
- **THEN** the extracted output contains compact event, card, session, fallback, and error summaries suitable for judge and frontend display

#### Scenario: Complex cards are normalized
- **WHEN** output includes card snapshots, card deltas, duplicate cards, or nested card metadata
- **THEN** extraction returns comparable compact card summaries without depending on raw payload shape or duplicate event order

#### Scenario: Fallback is boundary-aware
- **WHEN** output uses fallback or reports unavailable downstream data
- **THEN** judge normalization compares it with `allow_fallback`, expected stage, and current-case reference evidence instead of treating fallback as always correct or always incorrect

#### Scenario: Batch compact result is persisted to frontend
- **WHEN** batch status returns completed runs or case events
- **THEN** compact runs exclude `trace.raw_response` and `frontend_view.raw_sections`

#### Scenario: Required protocol evidence is missing
- **WHEN** a reference contract requires SSE events, stages, path types, or fallback behavior that the output does not satisfy
- **THEN** judge normalization records the missing or conflicting evidence and does not silently mark the case correct

### Requirement: Marketing-planning attribution is current-case evidence grounded
The system SHALL build attribution context from the current RunTrace, JudgeResult, reference contract, output summary, application boundary, and execution trace, and SHALL avoid failure attribution for correct verdicts.

#### Scenario: Correct verdict is attributed
- **WHEN** judge verdict is `correct`
- **THEN** attribute output does not invent a failed component or root cause

#### Scenario: Incorrect or uncertain verdict is attributed
- **WHEN** judge verdict is `incorrect` or `uncertain`
- **THEN** attribution identifies the earliest supported failing stage or reports insufficient evidence rather than using unrelated historical fields
