
# 背景：为什么需要多轮的特化
我现在看到的表格是什么样的，我希望它变成什么样
     
现在表格只有两列：Input 和 Output。每一行是一个 case，一次执行。一个 case 的 Input 是用户的一个请求，Output是系统对此的返回结果。然后 Judge 评分，Attribute 归因。

这个设计在单轮场景下很好——输入一次，输出一次，一目了然。

但真实业务中很多交互不是这样的。比如一个用户想"帮我找某个客户的所有下游关系"，这个意图不可能一次就输出完整的答案——系统需要先问"你要找哪个客户"，用户回复后，系统可能又问"关系类型是什么"，用户再回复……这种来回三到五轮的对话才算完成一个意图。

现在的问题是：这种多轮交互在表格里根本没法正确展示。如果我把每一轮都拆成一行，Input列就会到处重复出现同一个用户的原始意图，Output列也只能看单个轮次的孤零零的返回，你没法看懂整条链路到底发生了什么。到最后 Judge和归因也是基于某一轮的孤立输出在判断，信息不完整。

# 多轮的设计方案
所以我希望的表格是这样的：

Input 列：仍然只放一个东西——这个 trace 背后代表的是用户的哪一条原始意图。不管交互了多少轮，Input列始终是同一句话。这样一个 trace 在表格里就只有一行，用户一眼就能看出来这两行表格分别对应的是"查客户关系"和"查采购计划"这两个不同意图，而不是被多轮拆分乱了。

Output 列：这个列要能展开。默认收起的时候，只显示这条多轮交互的最终产出结果（最后一轮的系统返回）。但你需要能点进去，看到完整的交互过程——每一轮 mock 用户是怎么提要求的、系统是怎么回应的、下一轮 mock用户又基于什么来继续追问。这个展开过程是一个完整的对话链路，不是碎片化的几行。

Judge 和归因的粒度：基于整条 trace 来判断，不是只看最后一轮。因为用户意图是在整个多轮交互中逐步满足的，只看最后一轮的输出很容易误判——也许前三轮已经把上下文搭好了，第五轮才出结果。Judge和归因看的是整条链路是否忠实执行了用户意图。

Mock 用户的行为：现在 mock 用户一次给一个输入，系统一次给一个输出。多轮模式下，mock用户需要能像一个真实的人一样——看到上一轮系统的返回后，再决定下一轮要怎么追问。这个决策基于初始意图和已经交互过的上下文，不是预先写死的剧本。交互什么时候结束，由 mock用户判断（目标满足了就停）或者到一个上限（比如最多五轮），超过上限即使没完全满足也停止。

对表格展示的要求：这个展示逻辑要在 live 页和 summary 页都生效，不能只有某一边支持。summary里的聚合统计也照常按 trace 为单位，不因为某个 trace多轮就给它加权重或单独分类——轮次数是内部执行细节，对外部用户来说它就是一个目的、一次执行、一个评分。

# 目标
我提这个需求的核心目的是什么

我现在做归因和评估，最怕看到的信息不完整导致判断失真。如果你把一个五轮交互拆成五行表格，每行的上下文都是残缺的——Input 重复五次，Output 孤立五次，Judge看着第五轮的输出了一个结论，但实际上失败可能发生在第二轮。我需要能看到完整链路，才能判断得准。这才是为什么 I/O列的展示方式和 trace 的粒度这么重要。

这整个事情我说的是用户侧看到的表格长什么样和评估/归因基于什么粒度来判定这两件事。我没有在讲代码怎么写、字段怎么命名、接口怎么设计、pipeline 怎么路由——这些是后面根据这个需求推导出来的事。
