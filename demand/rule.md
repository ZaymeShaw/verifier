# 实现规范

1. ai编写内容，统一放在impl文件夹下
+ 通用的放在impl下按序构建。最好是构建一些协议类的东西，确定一个规范，包括judge、attribute、前端页面等，然后业务的具体实现按照协议来填这个框就行
+ 项目相关的，放在impl/projects下面的，归属于业务自己的实现，要去实现根据这个协议的具体内容点
+ 部分情况下需要构建一些模版XXX-template.md，放在impl下面，然后用户能够拷贝这些模版，填写模版里面的字段放在/projects/<project>/XXX-template.md，填写好后就能成为 <project>项目的具体标准，ai生成具体的东西落地在/impl/projects/XXX.md，然后具体的实现根据这个标准来进行调整
2. 协议实现tool模块，放在impl下，并为各项目在协议范围内实现个性化工具放在simpl/project/<project>/tools，供各项目智能使用工具。工具调用层目前使用agno来实现

# 核心概念
> 协议(最核心)
确定一个范围和标准，让模型在协议的范围内进行生成

>> 协议注意点
>> 协议涉及边界的部分怎么写
- 边界协议需要体现出真正重要的边界区分点
- 通用性：任何项目上，能能通过这套协议构建边界，而不是有的项目试用这套协议，有的项目不适用

>> 各个协议之间，必须得能串起来，然后协议的设计要有足够的灵活性，能支持简单的也能支持复杂的s

> 模版
>> 模版是什么
- 对于某<project>需要用户针对定向问题去补充解答的一组固定信息集合
- 各<project>针对此定向问题都有这组固定信息集合需要补充，因此具有通用性

>> 模版构建原则
+ 目标导向：填写每个模版，都是为了能让用户能通过明确一些东西，填写后就能针对该项目制定出明确的标准、划分好边界等
+ 通用性：任何项目上，能能通过这套模版构建项目对应的标准，而不是有的项目试用这套协议，有的项目不适用
+ 可填写性：模版得是用户能看懂，能填的。这届需要模版有一定的解释模版的成本，同时好填易填，要填的东西尽量不要太多，然后要填的东西应该字段上相对固定但是内容上能相对灵活让用户能好好的说明自己的理解，这个需要针对该模版的目标来调整（现在还是没看懂到底拷贝后用户要填啥）
+ 扮演好的引导：模版要填写的字段，应该要起到好的引导作用，能让用户填写好这些内容后，就能完成实现模版需要的目标，而不是简单的罗列一些东西让用户填，用户不知道咋填然后乱填然后填了也没作用
+ 模版本身提供的字段，就应该要起到一个提供信息量的作用，就是通过你给的要填的字段，用户应该能自然联想到哪些能区分好标准和边界上的东西，然后只要正常不用过多思考的填好，就能逐步让边界区分了，而不是给一个很含糊不知道该填什么好的东西，这样用户可能没有足够能力给出好的输入，模版也就失去了他的作用

> trace：业务系统的一次完整执行链路
- trace运行时：业务系统执行（可能伴随mock agent模拟用户）
- trace运行后：通常后接judge、attribute等agent分析



# 算法agent的约束

## 评估责任边界说明

> 为什么需要评估边界
有时候我们需要评估理想情况下应该做的怎么样，有时候我们想评估我们责任范围内做的怎么样，以下是两类典型情况
1. 理想情况：纯粹评估有没有完成好用户意图，不管实际条件如何（包括可能的上下文局限、系统局限等因素）
2. 现实情况：评估在当前系统条件下，完成了系统可做好范围内的多少（比如说在系统当前数据库配置等局限下，最多只能做到这样，即便不是完美完成用户意图，也属于一百分

简单来说就是，在我们评估同一个事情时，在不同的约束下，得出的结果也是不一样的
+ 在完全理想的情况，用户也没有给出具体约束时，我们以最理想的范围内评估，通过看回答能否满足用户意图为核心评判因素，能满足多少就是多少分
    - 典型场景：用户直接提供输入、模型输出。此时用户没有给出任何约束项，因此应以输出能否满足，满足多少用户意图为唯一判断标准
+ 在用户给出了系统的情况下，那我们应该在系统处理责任范围内进行评估
    - 比如回答不完全满足用户意图，但是不满足意图的卡点是因为系统之外的因素没做好导致，那这种就不是我们责任范围内，因此不应该以此扣分。评估系统仅评估系统范围内能做到多少分
    - 例子1：客户搜索下游数据库缺乏某关键字段导致无法搜索特定类型客户，这种情况不是客户搜索代码系统能优化的，因为用户数据存储不是客户搜索系统的责任范围，此时该问题不应该扣客户搜索结果的评分
    - 例子2：由于prompt编写错误、配置文件配的有问题、pipeline构建有问题、模型能力不够等因素，导致客户搜索接口没有产出正确结果，此时问题就是客户搜索的，应该扣他的分

> 怎么构建评估边界

>> 流程
+ ai构建impl/judge_boundary-template.md
+ 用户可以自行选择拷贝judge_boundary-template.md，填写责任边界信息后放到<project>/judge_boundary-template.md
+ analysis agent根据用户填写内容在协议内实现impl/project/的judge逻辑（请不要通过prompt的方式实现，而是通过流程的方式实现边界判断）

>> 注意点
+ 边界协议不需要填写所有judge需要的东西，只需要填写针对项目评估边界上面的东西，所以协议要填写的东西应该也只需要让用户围绕这方面填最小化的在各项目上可能变化的点
+ 边界是用来完善和进一步确定judge构建逻辑，请结合此对项目的judge进行优化，确保项目judge满足边界协议的要求，以及确保当前协议能满足judge的需要，必要的时候也要调整judge协议（不是说边界协议，是judge协议）
+ 应在协议中体现，分析系统是怎么样的来在协议范围内制定judge/attribute的impl或impl/project实现，而非每次由judge独立判断（注意这点）




## 算法优化要避免的东西

1. 如果要优化模型，比如避免过拟合等，尽量不要用规则类手段，遵循check.md
2. 过拟合的问题是有必要解决的，但是不能通过规则的方式处理，遵循check.md
