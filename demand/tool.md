# 怎么做tool

需要对原业务系统进行分析，解析出可能有用的源代码函数，构建为成测评系统的tool
大致可以理解，为各个项目选择合适的工具，并合理的分配在第一层和第二层中


## 合理tool分层

对，当前的关键问题是 tool 的**分层设计**没做好。应该分两层：

你说得对。我之前的"第一层=确定性校验、第二层=LLM深挖分析"划分是错的，本质上是把"我知道答案"和"我不知道答案"的边界硬套了一个哲学标签。

真正的分层应该是**获取信息的方式不同**：

**第一层（pipeline 预执行）**：信息通过工作流自动获取，确定性高、成本低、不需要 LLM 参与。比如 `get_runtime_checks` 直接调 INTENT_MAPPING 查表、`simulate_trace_nodes` 沿 trace 逐节点跑业务函数。这些信息是**固定流程产出**的，case 之间差异只体现在数值上。适合"每个 case 都需要的、可枚举的"信息获取。

**第二层（LLM 动态调用 tool）**：信息获取方式从"工作流"变成"动态决策"。LLM 根据当前 case 的上下文自主决定调哪个 tool、传什么参数、怎么组合。适合的场景是：
- 上下文太多，工作流无法全量塞进 prompt（需要 LLM 按需查询局部信息）
- 适合工作流获取但**组合方式**不是固定的（比如同一个 case 既要查字段定义又要查映射表又要对比 rules，顺序和组合 case 间不同）
- 需要结合 LLM 的推理能力**先判断缺什么再查什么**（而不是预先定义好"查 A 查 B 查 C"的流程）

换句话说，**不是"第一层负责简单问题、第二层负责复杂问题"，而是"第一层用固定流程获取信息，第二层用动态决策获取信息"**。同一个 case 两个层面可以获取同一类数据，只是方式不同——第一层是自动化管线，第二层是 LLM 按需探查。

这个划分和我之前给 tool 加 `root_cause_chain`/`fix_direction` 之类的结论判断没关系——tool 返回什么内容取决于"LLM 需要什么信息来做判断"，不是 tool 替 LLM 做判断。
