## 怎么做judge



> 重新理解输入背后的核心用户意图。评估业务服务输出是否cover用户需求（意图识别）
+ judge简单来说，就是在我们责任范围内，看我们的系统做的怎么样，看系统的输出对用户的意图完成的怎么样，无关乎具体实现。
+ attribute评估我们责任内，系统出问题的情况下分析全链路看具体是哪里有问题
> 根据业务系统能力边界的不同，评估系统也要进行相应调整，主要是根据业务系统的责任范围划定评估边界
+ analysis agent根据业务系统的责任边界，来确定评估系统的评估范围（judge agent的评估边界），并在impl/project里面实现
> 关于评估边界具体请看demand/rule.md

## 怎么做归因

> 归因要做到得东西
1. 以能否解决问题为导向：
    + 如果没有发现真正的问题，你应该要将其视为没有成功，然后继续优化。给出明确的修改方案，定位到具体的点（但是不直接改代码）
    + 有可追溯、可执行、可直接使用、采取后能直接解决问题的证据链路，确定链路中哪些正常哪些不正常
    + 归因总结的东西得是有用的，得让开发能知道具体问题点点，不要只是模棱两可的说是因为某个模块出问题了，我看了这种描述都不知道怎么修复
2. 手段：先将业务项目构建为trace，拆解进行分析
    + 将api链路拆解为可操作的trace，包括大的流程，以及小的一些关键函数组件，都记录在trace中，执行局部链路模拟测试脚本，验证根源问题发生点
    + 模拟trace分析尽量用现有的写好的项目代码，通过import之类的方式引用啥的，避免编写一些不存在的东西出来，最后得到的结论也与本项目无关
    + 看看在整个trace中，串到哪个点为止链路到底是不是正常的，从而识别出来有没有问题。做这种api测试/局部链路函数/模拟链路测试，看到哪里为止是通的，通过比如发现trace链路的某个节点上得到的结果与预期不符，得出结论所以最终问题是来自于哪里
3. 问题定位要保证：
+ 体现问题详细位置：要体现问题代码位置，指出具体哪个大模块中的哪个组件/函数与预期不符，预期是怎么样的实际时怎么样的。原因是哪个配置/prompt/代码链路的误差导致。后续该怎么优化
+ 指出问题有说服力：归因问题时必须去实际看代码/日志/模拟调用/试跑api/试跑源代码/试跑pipeline等，给出确实的可由其他人复现的证据，不要有任何主观臆测，未经验证不要做出判断
4. 以上设计原业务项目定位/拆解的部分，可以通过application实现原系统模拟使用工具，从而智能加载源代码/跑源函数等，从而支持归因



## 怎么做mock

部分信息：

> mock数据的初始化：
1. reference数据的生成必须得符合项目约束的。你需要你能获取到的所有项目信息来判断（包括模拟构建输出、查看源代码等手段）
2. 为各个场景从不同角度出发，构造有差异性的多个100条数据集，数据集之间彼此不相同，同数据集间也不相同，放在data/<project>目录下



# 算法agent设计

请通过协议实现如下：

1. 多步执行
2. 在约束下执行
3. 设计项目等部分，通过构建项目专属的tool等方式，来实现项目的个性化
4. 项目信息等庞大的上下文放在记忆中，直接复用agno等记忆机制来管理长上下文（judge等agent的记忆机制实现使用agno）
    - 不同project分别实现和接入自己的知识库，数据库本身放在impl/knowledge/<project>下，实现放在impl/projects/<project>，相关协议放在impl下
    - 各项目的知识库，请引用所需的长上下文信息，比如各种数据、字段定义、配置、文档、源代码信息等
    - 检索这种你没必要用关键词，就用语义检索之类的，直接复用agno
    - 记忆实际检索阶段，可以多放点，不用做的太尖，多放点信息
    - 涉及模型检索的部分，你可以调用百炼的向量模型，参考env.md


## agent触发
> analysis agent：<project>初始化/信息更新时触发，基于claude sub agent实现
> check agent：不定期触发，基于claude sub agent实现
> application agent：业务项目有更新时触发，基于claude sub agent实现
> build agent：<project>初始化/信息更新时触发，基于claude sub agent实现
+ mock agent：由agent脚本执行
    + 预构建批量数据时，批量构建意图/输入query信息时触发（此时不涉及实时多轮交互）
    + trace运行时，需要模拟用户时触发（通常涉及实时多轮交互）
+ judge agent：trace运行后，需要评估系统能力时执行，比如trace构建后需要评估触发，由agent脚本执行
+ attribute agent：trace运行后，judge判断业务系统执行不及预期时执行，比如trace构建有需要分析问题时触发，由agent脚本执行

## 各agent注意事项

> trace分析时，（judge、attribute、mock等）的分析尽量是中文
> trace分析时，实时调用大模型分析，不应该是秒出结果
> trace分析时，请调用deepseek-v4 pro，启动max思考模式，根据目标进行拆解，实时多轮深度分析（参考env.md）
