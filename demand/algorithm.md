




# 信息密度原则：
1. 不损失、遗漏有效信息
2. 不引入无效、偏差、误导性信息
3. 奥卡姆剃刀：维持相同有效信息量的情况下，方案越简洁越好，信息总量越少越好
4. 结合上述，保证最高的信息密度


# 系统信息量评估
评估当前算法方案的上下文工程，对应的信息量的有效性

报告的受众是人类，或者ai，通过阅读这篇报告，评估某个信息量是否有价值，是否遗漏，来评估当前上下文工程是否合理，是否应该填补某上下文或者去掉冗余上下文

我觉得要做目标分解和系统分解，把元素先拆出来，然后看看对应的上下文信息当前上下文工程能不能搞出来，不搞出来的话对于完成目标信息量损失多少

基于目标，对系统可获取上下文信息进行分解，列出所有信息量，看各信息量对于完成目标的有效性评分
1. 以业务目标为导向，阅读业务系统文档/代码，理解完成业务目标所需的核心信息
2. 分析评测系统可获取的所有业务系统信息量，基于此范围列出信息量
2. 对业务用户需求进行分析拆解，对每项信息量进行评估，分析各信息量的评分值

|信息量|测评系统是否可获取|当前上下文工程对该信息量的覆盖率|有效性评分|信息损失率|关键损失信息|当前上下文展示真实示例（截取样本prompt获取，给出的内容是prompt的一部分）|
|---|---|---|---|---|---|---|
|根据系统信息和用户意图分解|是/否|0-100|0-100|0-100|...

结论：评估当前上下文工程是否合理，是否应该填补某上下文或者去掉冗余上下文，从业务角度说明
