本项目的核心目标是从case分析的角度，对业务系统进行模拟交互、输出评估、问题归因，从而看到业务系统当前有哪些问题（包括业务系统功能、算法能力，特别是算法能力方面），进而改进项目
需求实现请遵循demand/rule.md的约束，在约束特别是协议的范围内进行实现，任何修改先验证是否与协议对齐。如果需要修改协议需要全部项目一起对齐，工程量较大，需谨慎并由用户确认
启动本评测项目时请触发所有业务项目的start.md启动项目api
环境相关问题可以看env.md

# 需要拥有的能力

1. 业务和代码理解能力：从api文档中，加载api信息，理解业务背景，解析理解代码逻辑
2. 构建虚拟测试数据，基于业务背景和项目信息，构建或加载输入测试数据/意图
3. 应用执行能力：启动上下文无依赖的独立的环境，构建并启动模拟api服务或者启动/加载现有api服务
3. 批量运行能力：实时或批量运行项目api，获取api输出结果
4. 判断能力：判断api输出是否正确。说明为什么该输出不及业务预期
5. 归因能力：当输出不及预期时，能解析项目代码流转逻辑的情况下，清晰的指出不及预期的原因，并且为开发提供切实明确有意义的优化方案，归因总结的东西得是有用的，得让开发能知道具体问题点，不要只是模棱两可的说是因为某个模块出问题了，这种描述看了都不知道怎么修复
6. 代码标准/一致性审查：定期检查ai生成代码，保持一致性，对齐项目各处代码/pipeline/agent/数据等的逻辑和标准


# 需要拥有的agent

1. analysis agent：从projects/<project>中(由用户编写基础项目信息，以及对各个agent对补充要求)，读取相关文档信息，抽取项目api、业务背景信息，抽取项目相关信息放到impl/projects文件夹下，包括
    + 理解api调用链路，构建接口文档、接口格式yaml文件等（部分没有完整项目的场景，需要根据application agent生成情况进行调整）
    + 如何mock数据集，构建独立mock方案文档
    + 如何适配前端协议构建合理前端，抽取前后端实现架构文档
    + 理解如何对api输出结果进行评判，分析得出评判方案标准文档、评估agent脚本等
    + 理解整个代码运转链路pipeline和关键配置文件、核心函数等，抽取问题归因方案文档、归因agent文档等
2. check agent：对代码进行审查，给用户进行核对，用户确认后，对general_eval项目代码进行更新
3. application agent：根据接口文档，启动业务项目服务
    + 在有现成项目等情况下，启动项目api服务/脚本/pipeline等，形成标准化使用方式
    + 在当前没有现成完整项目的情况下（比如仅有skill），模拟构建agent链路，形成脚本、api服务、pipeline，构建可落地的代码文件，形成标准化使用方式
    + 自测验证标准化使用方式正常可用，将标准化使用方式放在application文件夹中
4. build agent：
    + 结合analysis agent的分析，构建项目前端
    + 根据analysis agent的分析，为各项目构建mock、judge、attribute所需要的tool，放在impl/project/<project>/tools中，供mock、judge、attribute按需调用
4. mock agent：结合mock方案文档，构建虚拟测试数据，包括 ：
    + 模拟构建用户意图
    + 基于用户意图和当前应用上下文（系统交互），模拟用户下一步输入（可适用单论/多轮交互场景）
5. judge agent：结合评判标准文档，判断一个session中的链路输出是否正确、得出评分等
    + 目标分解：分解本项目上做好judge需要的工作，在impl/project构建能力强的judge agent（项目初始化时触发）
    + 意图识别与评估：judge本质是对外部期望的感知，judge是业务目标导向的，他要判断本业务系统的输出能否满足外部/下游系统期望的要求，也就是需要judge判断本项目外部下游依赖方是谁/哪个系统、外部依赖需要的格式是什么、意图是什么，通过判断是否满足外部依赖的预期来评估得分（trace运行后触发）
        - 重建意图：不要复用业务系统给出的意图，judge agent需要重建用户准确意图
        - 准确、全面的识别用户对系统的完整期望，不要缺、漏、错
        - 理解系统的能力边界，做出合理判断
        - 评估尽量完全从业务的角度和业务的用词出发，从具体的输出字段内容关联到业务概念上分析是否符合预期
    + 工具调用：通过使用外部依赖/意图等相关工具，更好的抽取意图和系统边界，得出正确的判断
6. attribute agent：当输出不及预期时，能结合归因方案文档，从业务项目代码链路中归因出不及预期原因
    + 目标分解：分解本项目上做好attribute需要的工作，在impl/project构建能力强的attribute agent。（项目初始化时触发）
    + 智能触发：judge判断业务系统结果不及预期时执行，比如业务系统输出被判定为错误/分数较低时执行
    + 代码分析：attribute归因，相比judge要多出一个业务系统代码的输入，judge只看业务系统的输入输出，而attribute还会看业务代码，attribute需要从系统代码中分析问题发生的原因
    + 内审定位：attribute归因相比judge更偏重于内部系统的管理与技术内审
    + 工具调用：通过使用引用原业务项目内容构建的tool（尽量直接import原项目函数调用），辅助归因业务系统的问题
