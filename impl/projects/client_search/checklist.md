# Check List

Check this project through the generic protocol chain:

1. The 8000 business API is reachable, or the run is explicitly marked mock/error.
2. `RunTrace.raw_response` preserves the original response.
3. `RunTrace.extracted_output` contains a generic summary and structured output.
4. client_search-specific fields remain in `project_fields` or raw response.
5. judge uses the current trace, `evaluation.md`, implementation `judge_boundary_protocals.md`, project judge standard `judge.md`, the user-owned `projects/client_search/judge_boundary-template.md`, and the latest user-owned `projects/client_search` prompt/config references.
6. user-facing judge boundary template lives at `impl/judge_boundary-template.md`; `projects/client_search/judge_boundary-template.md` is the user-filled project standard, and `impl/projects/client_search/judge_boundary_protocals.md` is the AI implementation mapping.
7. the user-filled judge boundary remains a plain-language final-verdict standard; internal JudgeResult boundary fields are generated by the judge/check pipeline.
8. attribute uses the current trace, judge result, `attribution.md`, current prompt/config/start evidence, and executable trace nodes when available.
8. frontend displays project fields only as extensions/raw sections.
9. generic core does not hardcode client_search field names, ports, or cases.
10. failures include evidence, verification steps, and patch direction before being considered useful.
11. live frontend keeps the single-case flow to business request, judge, and attribute; cluster/check remain summary-level concerns.
12. project implementation docs are resynced when user-owned `projects/client_search` docs change.
13. failure/uncertain attribution should mark which execution trace stages are normal, suspicious, failed, or not yet verified.
