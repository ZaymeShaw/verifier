# Judge Boundary Template

这个模板只解决一个目标：帮助 judge 区分“业务输出错误”还是“系统能力边界导致无法满足”。

用户复制到 `projects/<project>/judge_boundary-template.md` 后，只填写项目评估边界中真正会变化的内容。不要在这里写完整 judge 规则、接口字段全集、`JudgeResult` 字段或归因方案；这些放在项目 readme/config/prompt/evaluation/judge 文档里。

## 填写方式

围绕下面 3 个问题填写：

1. **限制是什么**：项目有哪些不是本系统能直接决定的上游、数据、字段、枚举、环境或业务限制？
2. **哪些仍要评价**：在这些限制之外，哪些部分是本系统可优化、可修复、可控制的，因此做错要判 `incorrect`？
3. **依据从哪来**：judge 应该从哪些资料确认限制和可控范围？资料冲突时谁优先？

## 模板

```markdown
# Judge Boundary - <project>

限制：<写清楚哪些用户需求可能因为上游依赖、数据库、外部系统、权限、字段/枚举缺失等原因无法完全满足；这些不属于当前系统可直接控制的输出错误。>

评价范围：<写清楚哪些部分仍属于当前系统能力范围，做不好应纳入评估并可能判 incorrect。比如模型理解、项目配置、prompt、业务代码、后处理、字段映射等。>

边界依据：<写 judge 应该从哪些项目资料确认限制和评价范围；如果资料冲突，写优先级。>
```

## 填写检查

填完后应该能判断：

- 一个没满足的用户需求，是系统输出错误，还是当前系统无法控制的能力差距？
- 如果是模型、配置、prompt、代码、后处理能优化的问题，是否会纳入评分？
- judge 凭什么资料确认某个字段、枚举、数据库能力或外部依赖是否存在？

## 示例

```markdown
# Judge Boundary - client_search

限制：有些用户需求受限于当前上游依赖的 ES 数据库中没有相应字段或枚举值，可能无法完全满足用户意图搜索出正确字段；这类限制按业务系统能力边界判定，不直接视为当前系统输出错误。

评价范围：如果是当前系统可以优化的部分做得不好，比如大模型理解、项目配置、prompt、项目代码、字段映射或后处理错误，就应该纳入衡量范围，按实际错误判定。

边界依据：数据库和字段/枚举相关信息从 `projects/client_search/readme.md`、`config.md`、`prompt.md` 和原项目 ES 信息中确认；资料冲突时，优先相信最新 config 和 prompt。
```
